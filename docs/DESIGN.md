## Problem Statement

Two podcasters need a simple web platform to record conversations together without external audio equipment or complex setup. Both users log in, enter a shared room, connect via audio (no video), press a single red button to start recording, and after the session ends, both can download the same mixed recording file.

## What Makes This Cool

One button. Two voices. One recording file. No equipment, no setup, no post-session file merging. The entire recording experience fits in a single web page after login.

## Constraints

- 2-week timeline
- 2 users maximum (no multi-room scaling needed initially)
- MP3 192kbps or better audio quality
- Server-side storage for recordings (file system, HTTP download)
- Simple database auth (SQLite + bcrypt + sessions)
- Next.js only — no separate Go backend, no separate signaling service

## Premises

1. Two users only — just the two founders, no scale needed initially
2. Next.js only (API routes + SPA) — one runtime handles everything
3. Browser-to-browser WebRTC (p2p) — audio goes directly between peers, server only signals
4. Local browser recording via MediaRecorder — higher quality, simpler than server-side recording
5. Server-side FFmpeg mix (one-shot) — after both browsers upload recordings, server mixes once
6. 2-week timeline — ship working, not perfect

## Approaches Considered

### Approach A: Go backend (pion) + Next.js SPA (original)

- Summary: Go backend (pion/webrtc) handles auth, signaling, WebRTC audio, FFmpeg recording; Next.js SPA handles login/room/download UI
- Effort: M
- Risk: Low-Med (pion audio-to-FFmpeg pipeline is proven, ~20 lines)
- Pros: pion gives raw PCM directly from WebRTC audio tracks. FFmpeg integration is simple. No C++ build complexity.
- Cons: Two runtimes (Go + TypeScript). Slightly more deployment complexity.

### Approach B: Decoupled backend + frontend

- Summary: Separate Go/Node.js backend for signaling + recording, Next.js SPA frontend
- Effort: L
- Risk: Med
- Pros: Clean separation, extensible to more users/rooms
- Cons: Two services, 2 weeks is tight for both, CORS/auth overhead

### Approach C: Browser-only recording with server mixing (CHOSEN)

- Summary: Next.js only. Browser-to-browser WebRTC for live audio. Each browser records locally via MediaRecorder. After session, both upload, server mixes once with FFmpeg.
- Effort: S
- Risk: Low
- Pros: One runtime. No server-side media handling during call. Better audio quality (direct mic capture, no network loss). No pion complexity. No live FFmpeg pipeline. Simple WebSocket signaling.
- Cons: Need to upload files after session (small delay before mixed file available). Users need network connectivity during recording for live audio (WebRTC p2p).

## Recommended Approach

Approach C. Next.js only, WebRTC p2p for live audio, local browser recording, one-shot FFmpeg mix on server. Simplest approach for 2 weeks, 2 users. KISS, DRY, YAGNI.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    Next.js (one runtime)                     │
│                                                               │
│  Pages (SPA):                                               │
│    /login          — Login / Register                        │
│    /room           — Shared room, WebRTC, record button      │
│    /download       — List recordings, download links          │
│                                                               │
│  API Routes:                                                │
│    /api/auth       — Register, login, logout (bcrypt)         │
│    /api/ws         — WebSocket signaling (SDP/ICE exchange)   │
│    /api/upload     — Receive uploaded recording blobs         │
│    /api/mix        — Trigger FFmpeg mix → output MP3          │
│    /api/recordings — List/download recording files            │
│                                                               │
│  Database: SQLite (better-sqlite3, sync, no pool needed)     │
│  Auth: bcrypt + cookie-based sessions                         │
│  Signaling: WebSocket (ws library, Next.js API route)        │
│  Storage: Local filesystem (./recordings/)                    │
└─────────────────────────────────────────────────────────────┘

WebRTC Audio Path (p2p):
  Browser A ──(SRTP)──► Browser B    (direct, no server relay)
  Browser B ──(SRTP)──► Browser A    (direct, no server relay)

Signaling Path (via server):
  Browser A ──WS──► /api/ws ──WS──► Browser B  (SDP/ICE only)

Recording Path (local → upload → mix):
  Browser A ──MediaRecorder──► blob ──upload──► /api/upload
  Browser B ──MediaRecorder──► blob ──upload──► /api/upload
                                                     │
                    FFmpeg ──► MP3 ◄────────────────┘
```

### Tech Stack

- **Full stack:** Next.js (App Router, API routes, SPA pages)
- **Database:** SQLite (via `better-sqlite3` — sync, no connection pool needed)
- **Auth:** bcrypt + cookie-based sessions
- **Signaling:** WebSocket (`ws` library in Next.js API route). Single WS connection carries both auth messages and WebRTC signaling (SDP/ICE). Message type field distinguishes auth vs signaling.
- **WebRTC (client):** Native browser WebRTC API (RTCPeerConnection, getUserMedia)
- **Recording (client):** Native MediaRecorder API (built into browser, no library)
- **Mixing (server):** FFmpeg (one-shot command: mix two uploaded files into one MP3)
- **STUN:** Google's public STUN servers (stun.l.google.com:19302) — free
- **TURN:** coturn self-hosted on the same VPS (only needed if peers are behind restrictive firewalls — defer to post-launch)
- **Storage:** Local filesystem (`./recordings/`)
- **Styling:** Minimal CSS (no framework needed)

### Live Audio + Recording Pipeline (simplified)

```
Live Audio (during call):
  Browser A ──(WebRTC p2p)──► Browser B    (hear each other)
  Browser B ──(WebRTC p2p)──► Browser A    (hear each other)

Recording (during call):
  Browser A ──MediaRecorder──► local blob  (direct mic capture)
  Browser B ──MediaRecorder──► local blob  (direct mic capture)

Mixing (after call):
  Upload A ──► /api/upload ──┐
                              ├──► FFmpeg ──► session-{timestamp}.mp3
  Upload B ──► /api/upload ──┘
```

1. Both users join `/room`, WebSocket connects them via `/api/ws`
2. WebRTC peer-to-peer connection established (server only passes SDP/ICE)
3. Both hear each other live through browser speakers/headphones
4. Either user presses red record button → 3-second countdown → both start MediaRecorder
5. Either user presses "Stop" → both stop MediaRecorder, get final blobs
6. Both blobs uploaded to `/api/upload` (via FormData)
7. Server spawns FFmpeg: `ffmpeg -i track-a.webm -i track-b.webm -filter_complex amix=inputs=2:duration=longest -b:a 192k output.mp3`
8. Final MP3 saved to `./recordings/session-{timestamp}.mp3`
9. Download page updated, both users can download

## Open Questions

- TURN server: Only needed if peers are behind restrictive firewalls. Can defer to post-launch if both users are on typical home networks.
- Recording file naming: `session-{ISO-timestamp}.mp3`
- Upload format: MediaRecorder outputs webm/opus — FFmpeg handles this input natively
- CORS: Not needed — Next.js API routes and SPA share the same origin (one Next.js server)

## Error Handling & Edge Cases

- **Peer disconnects mid-session**: Recording continues. Reconnected peer resumes audio. If peer never reconnects, recording ends when other user stops.
- **Server crash during mixing**: Partial MP3 file is saved but may be corrupted. On restart, user sees partial file and can delete/re-record.
- **Session lifecycle**: Session ends when either user clicks "Stop" or both disconnect for >60 seconds. Recording auto-stops.
- **Concurrency**: Single room hardcoded. A third user attempting to join sees "Room full."
- **Disk full**: Server checks available disk space before starting mix. Fails gracefully with error message if <100MB free.
- **WebSocket reconnect**: If signaling drops, peers attempt ICE reconnection. After 3 failures, room page reloads.
- **Upload failure**: If one browser fails to upload, server retries once. If still fails, recording is saved with single track only and user notified.

## Success Criteria

1. Two users can log in and join the same room
2. Audio connection works with minimal latency (<500ms)
3. One red button starts recording for both sides
4. After session ends, both users can see and download the mixed MP3 file
5. Audio quality is clear enough for post-production editing (192kbps MP3)

## Distribution Plan

Self-hosted deployment. Deploy on a VPS (DigitalOcean, Railway, or similar). Next.js runs as a single Node.js server. Can use nginx as reverse proxy. FFmpeg must be installed on the server. SQLite file is bundled with the app.

## Dependencies

- FFmpeg must be installed on the server
- Node.js 20+ on the server
- SQLite (bundled via better-sqlite3, no separate installation needed)
