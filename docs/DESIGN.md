## Problem Statement

Two podcasters need a simple web platform to record conversations together without external audio equipment or complex setup. Both users log in, enter a shared room, connect via audio (no video), press a single red button to start recording, and after the session ends, both can download the same mixed recording file.

## What Makes This Cool

One button. Two voices. One recording file. No equipment, no setup, no post-session file merging. The entire recording experience fits in a single web page after login.

## Constraints

- 2-week timeline
- 2 users maximum (no multi-room scaling needed initially)
- MP3 192kbps or AAC 128kbps audio quality
- Server-side storage for recordings (file system, HTTP download)
- Simple database auth (SQLite + bcrypt + sessions)
- Custom WebRTC signaling (not LiveKit)

## Premises

1. Two users only — just the two founders, no scale needed initially
2. pion/webrtc in Go for WebRTC — simpler and proven for recording pipeline
3. Server-side recording — server mixes and records both streams during the call
4. 2-week timeline — ship working, not perfect
5. Go backend + Next.js SPA — two runtimes, clean separation

## Approaches Considered

### Approach A: Go backend (pion) + Next.js SPA (CHOSEN)

- Summary: Go backend (pion/webrtc) handles auth, signaling, WebRTC audio, FFmpeg recording; Next.js SPA handles login/room/download UI
- Effort: M
- Risk: Low-Med (pion audio-to-FFmpeg pipeline is proven, ~20 lines)
- Pros: pion gives raw PCM directly from WebRTC audio tracks. FFmpeg integration is simple. No C++ build complexity.
- Cons: Two runtimes (Go + TypeScript). Slightly more deployment complexity than single monolith.

### Approach B: Decoupled backend + frontend

- Summary: Separate Go/Node.js backend for signaling + recording, Next.js SPA frontend
- Effort: L
- Risk: Med
- Pros: Clean separation, extensible to more users/rooms
- Cons: Two services, 2 weeks is tight for both, CORS/auth overhead

### Approach C: Browser-only recording with server mixing

- Summary: Each browser records locally via MediaRecorder, uploads after session, server mixes
- Effort: M
- Risk: Med
- Pros: Best audio quality, no server FFmpeg during call
- Cons: Clock sync complexity, delayed file availability, upload fragility

## Recommended Approach

Approach A. Go backend (pion/webrtc) for real-time audio + recording, Next.js SPA for UI. Right scope for 2 weeks, 2 users.

## Architecture

```
┌─────────────────────────────────────┐        ┌─────────────────────────┐
│         Go Backend (pion)           │        │      Next.js SPA        │
│                                     │        │                         │
│  Auth:                              │        │  Pages (static SPA):    │
│    SQLite + bcrypt + sessions       │        │    /login               │
│    Cookie-based auth (sets/validates)        │    /room                │
│                                     │        │    /download            │
│  WebRTC:                            │        │                         │
│    Peer A ──(SRTP)──► pion ──┐      │◄──────►│  Next.js calls Go APIs  │
│    Peer B ──(SRTP)──► pion ──┤      │  HTTP  │  (no server-side auth)  │
│                              ├──────► WS     │                         │
│    Audio tracks ──► raw PCM ──► FFmpeg        │                         │
│                              │                │                         │
│    Signaling: gorilla/ws      │                │                         │
│    File serving: HTTP         │                │                         │
│    Recordings: ./recordings/  │                │                         │
└─────────────────────────────────────┘        └─────────────────────────┘
```

### Tech Stack

- **Backend:** Go (pion/webrtc for WebRTC, gorilla/websocket for signaling)
- **Frontend:** Next.js (App Router, SPA for room + download pages)
- **Database:** SQLite (via `mattn/go-sqlite3` on backend)
- **Auth:** bcrypt + cookie-based sessions (set by Go backend)
- **Signaling:** WebSocket (gorilla/websocket, embedded in Go server). Single WS connection carries both auth messages and WebRTC signaling (SDP/ICE). Message type field distinguishes auth vs signaling.
- **WebRTC (client):** Native browser WebRTC API (RTCPeerConnection, getUserMedia)
- **WebRTC (server):** pion/webrtc (Go library) — receives SRTP, provides raw PCM from audio tracks
- **Recording:** FFmpeg (Go spawns FFmpeg subprocess, pipes PCM from pion audio tracks)
- **STUN:** Google's public STUN servers (stun.l.google.com:19302) — free
- **TURN:** coturn self-hosted on the same VPS (only needed if peers are behind restrictive firewalls)
- **Storage:** Local filesystem (`./recordings/`)
- **Styling:** Minimal CSS (no framework needed)

### WebRTC + FFmpeg Pipeline (the hard part, simplified)

```
Browser A ──(SRTP)──┐
                     ├──► pion/webrtc (Go) ──► audio track reader ──► raw PCM
Browser B ──(SRTP)──┘                                                │
                                                                     ▼
                                                              FFmpeg ──► MP3 file
```

1. Each browser creates an RTCPeerConnection with audio-only track from getUserMedia()
2. Browser sends SDP offer/answer + ICE candidates via WebSocket signaling
3. pion/webrtc establishes WebRTC peer connections with both browsers
4. pion reads raw PCM from each audio track via `track.Read()` method
5. Go mixes both PCM streams (simple interleaving) and pipes to FFmpeg's stdin
6. FFmpeg encodes to MP3 192kbps and writes to `./recordings/session-{timestamp}.mp3`
7. On stop, Go sends SIGTERM to FFmpeg, which finalizes the MP3 file
8. Go creates a database record linking the file to the session

## Open Questions

- TURN server: Only needed if peers are behind restrictive firewalls. Can defer to post-launch if both users are on typical home networks.
- Recording file naming: `session-{ISO-timestamp}-{room-id}.mp3`
- Recording trigger: Either user pressing record starts for both (per original spec). A 3-second countdown prevents accidental starts.
- CORS: Go backend and Next.js SPA may need CORS configuration during development. Can use Next.js proxy or set CORS headers on Go server.

## Error Handling & Edge Cases

- **Peer disconnects mid-session**: Recording continues. Reconnected peer resumes audio. If peer never reconnects, recording ends when other user stops.
- **Server crash during recording**: FFmpeg process is killed. Partial MP3 file is saved but may be corrupted. On restart, user sees partial file and can delete/re-record.
- **Session lifecycle**: Session ends when either user clicks "Stop" or both disconnect for >60 seconds. Recording auto-stops.
- **Concurrency**: Single room hardcoded. A third user attempting to join sees "Room full."
- **Disk full**: Server checks available disk space before starting recording. Fails gracefully with error message if <100MB free.
- **WebSocket reconnect**: If signaling drops, peers attempt ICE reconnection. After 3 failures, room page reloads.

## Success Criteria

1. Two users can log in and join the same room
2. Audio connection works with minimal latency (<500ms)
3. One red button starts recording for both sides
4. After session ends, both users can see and download the mixed MP3 file
5. Audio quality is clear enough for post-production editing (192kbps MP3)

## Distribution Plan

Self-hosted deployment. Deploy on a VPS (DigitalOcean, Railway, or similar). Go binary is a single static binary. Next.js runs as a Node.js server. Can use nginx as reverse proxy for both.

## Dependencies

- FFmpeg must be installed on the server
- Go 1.21+ on the server
- Node.js 20+ on the server (for Next.js)
- SQLite (bundled via go-sqlite3 CGO)
