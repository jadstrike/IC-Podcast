Primitives signatures (from PATTERNS.md):

Button:
```
type ButtonVariant = 'default' | 'primary' | 'ghost'
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: ButtonVariant
  block?: boolean
}
```
Renders `<button>` with className composed of `'btn'`, plus `'primary' | 'ghost'`, plus `'block'`. Class names verbatim.

Field:
```
interface FieldProps {
  label: string
  id?: string
  type?: React.HTMLInputTypeAttribute
  placeholder?: string
  autoComplete?: string
  name?: string
  defaultValue?: string
  required?: boolean
}
```
Renders `<div class="field"><label htmlFor>...</label><input .../></div>`. Use uncontrolled inputs with `name` so the parent form reads via `e.currentTarget.elements`.

Avatar:
```
interface AvatarProps {
  initials?: string
  empty?: boolean
  status?: 'online' | 'pending' | 'offline' | null
  size?: 'sm' | 'md'   // sm = 72px (room panel), md = 88px (lobby) — default md
}
```
Renders `<div class="avatar [empty]" [data-size="sm"]>{initials}<span class="status online|pending" /></div>`. Size override: when `size === 'sm'`, apply inline style `{ width: 72, height: 72 }` OR add a data-size attribute matched by a globals.css rule already present (`.panel .avatar { width: 72px; height: 72px }`). Choose the data-size approach — write a new selector `.avatar[data-size="sm"]` only if the existing `.panel .avatar` cascade isn't sufficient. Since LobbyView renders the default 88px and RoomView wraps avatars in `.panel`, the existing cascade is sufficient — `size` prop is informational only and can be ignored in this plan if unused.

LoginView tab state: `useState<'login' | 'register'>('login')`. Tabs row toggles state. Conditionally render the login form or register form based on state. Each form has its own handler.

LobbyView callbacks: accept optional `onJoinRoom?: () => void` and `onLeave?: () => void`. Default `onJoinRoom` to `router.push('/room')` so the page works without a prop. Default `onLeave` to `router.push('/login')`.
